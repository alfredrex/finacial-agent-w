# 一、项目最终定位

本项目不要定位成普通金融聊天机器人，也不要定位成单纯 RAG 项目。

最终定位为：

> 面向个人投资者 / 初级投研分析师的可信金融投研 Agent。
> 系统能够结合用户画像、财报结构化指标、研报/公告/政策文本、行情数据，完成精确问答、财报分析、公司对比、风险归因和个性化投研报告生成。

核心技术特点：

1. **KVStore** 负责用户记忆、状态缓存、短期数据缓存和索引指针。
2. **SQL FactStore** 负责财报核心指标等结构化、可计算、可校验事实。
3. **RAG / ChromaDB** 负责研报、政策、公告文本、管理层讨论、风险提示等长文本证据。
4. **Web/API Tool** 负责最新新闻、行情、公告、临时缺失资料的在线获取。
5. **Query Router** 负责根据用户问题选择正确链路。
6. **Verifier** 负责检查最终答案中的数字、单位、期间、来源是否一致。

------

# 二、核心架构目标

目标架构如下：

```text
用户问题
  ↓
Query Router
  ↓
根据问题类型选择链路
  ├── 用户画像 / 偏好 / 关注列表
  │       -> KVStore
  │
  ├── 实时行情 / 最新新闻 / 最新公告
  │       -> Web/API Tool
  │       -> KVStore 短期缓存
  │
  ├── 精确财报指标 / 同比环比 / 多公司对比
  │       -> SQL FactStore
  │       -> 缺失时触发按需解析 / 入库
  │
  ├── 研报观点 / 政策影响 / 风险提示 / 经营原因
  │       -> RAG / ChromaDB
  │
  ├── 综合投研分析
  │       -> SQL FactStore + RAG + KVStore + Web/API
  │
  └── 最终答案
          -> Verifier
          -> QAAgent / ReportAgent 输出
```

重点原则：

> 不要让 RAG 回答精确财报数字。
> 不要让 KVStore 承担长期结构化财务数据库职责。
> 不要让大模型直接猜财报指标。
> 数字类问题必须优先走 SQL FactStore。
> 解释类问题才走 RAG。
> 最新变化类问题走 Web/API。
> 用户偏好和缓存走 KVStore。

# 三、KVStore、SQL、RAG 的职责边界

## 3.1 KVStore 负责什么

KVStore 继续保留，并且仍然是项目亮点之一，但职责需要重新定义。

KVStore 适合保存：

### A. 用户长期记忆

```text
memory:user:{user_id}:profile
memory:user:{user_id}:watchlist
memory:user:{user_id}:risk_preference
memory:user:{user_id}:investment_horizon
memory:user:{user_id}:answer_style
```

示例：

```json
{
  "risk_level": "中低风险",
  "preferred_industries": ["消费", "医药"],
  "investment_horizon": "long_term",
  "watchlist": ["600519", "000858", "002594"]
}
```

### B. 会话状态

```text
session:{session_id}:recent_tickers
session:{session_id}:last_question_type
session:{session_id}:last_report_period
```

### C. 短期缓存

```text
cache:quote:{ticker}
cache:news:{ticker}
cache:index:{index_code}
cache:announcement:{ticker}:latest
```

如果底层 KVStore 暂时没有原生 TTL，不要重构底层。
在 value 中加入逻辑 TTL：

```json
{
  "data": {
    "price": 1445.0,
    "change_pct": 1.25
  },
  "meta": {
    "created_at": 1778577000,
    "expire_at": 1778577300,
    "source": "quote_api",
    "data_type": "quote"
  }
}
```

读取时由 CacheService 判断是否过期。

### D. 索引指针

```text
index:stock:{ticker}:latest_report
index:stock:{ticker}:rag_docs
index:doc:{doc_id}:parse_status
index:doc:{doc_id}:source_path
```

### E. Pipeline 状态

```text
state:ingestion:last_run
state:doc:{doc_id}:parse_status
state:doc:{doc_id}:error_count
```

------

## 3.2 KVStore 不应该负责什么

不要继续把 KVStore 作为财报指标的最终权威存储。

当前 L3 已经把 27 项财务指标存在 KVStore 中，这个可以短期保留，但后续要改成：

```text
SQL FactStore = 财报指标 source of truth
KVStore L3 = 股票基础信息 + 热点缓存 + SQL 查询结果缓存 + 文档索引
```

短期迁移策略：

1. V1 阶段可以保留 KVStore 中已有财务指标。
2. 新增 SQL FactStore 后，财务指标写入 SQL。
3. 查询时优先 SQL。
4. KVStore 中财务指标只作为兼容层或缓存。
5. 后续逐步去掉 KVStore 作为财务事实主存储的角色。

------

## 3.3 SQL FactStore 负责什么

SQL FactStore 本质上就是结构化金融事实数据库。

适合保存：

```text
营业收入
营业成本
归母净利润
扣非归母净利润
经营活动现金流净额
总资产
总负债
所有者权益
资产负债率
毛利率
净利率
ROE
EPS
销售费用
管理费用
研发费用
财务费用
存货
应收账款
分红
估值快照
历史行情日线
```

SQL 负责：

1. 精确查询。
2. 多公司对比。
3. 多季度趋势。
4. 同比 / 环比计算。
5. 排行榜。
6. 筛选。
7. 可溯源引用。
8. 验证答案数字是否正确。

------

## 3.4 RAG 负责什么

RAG / ChromaDB 不负责精确数字查询。

RAG 负责：

```text
研报全文
公司年报中的管理层讨论与分析
季报文字说明
风险提示
政策文件
行业研究
产业链逻辑
业绩说明会内容
投资者关系活动记录
历史行情复盘文本
技术指标解释文档
估值方法说明
```

RAG 适合回答：

```text
为什么公司毛利率下降？
管理层如何解释收入增长？
券商怎么看公司的未来增长？
行业政策有什么影响？
公司主要风险是什么？
白酒行业增长放缓的原因是什么？
经营现金流和净利润背离说明什么？
```

RAG chunk 必须带 metadata：

```json
{
  "ticker": "002594",
  "company_name": "比亚迪",
  "doc_id": "002594_2026Q1",
  "doc_type": "quarterly_report",
  "report_period": "2026Q1",
  "section": "management_discussion",
  "page": 8,
  "source_path": "data/reports/2026Q1/002594_比亚迪_2026Q1.pdf"
}
```

查询 RAG 时必须尽量加 metadata filter，不要全库无差别搜索。

# 五、财报入 SQL 的难点与解决方案

## 5.1 难点一：指标名称不统一

不同公司可能使用不同名称：

```text
营业收入
营业总收入
一、营业收入

归母净利润
归属于上市公司股东的净利润
归属于母公司所有者的净利润
归属于母公司股东的净利润
```

解决方案：

1. 建立 `metric_dictionary`。
2. 入库时保存：
   - `metric_code`
   - `metric_name`
   - `raw_metric_name`
3. 查询时永远查 `metric_code`。
4. 原始名称只用于溯源。

示例：

```text
raw_metric_name = 归属于上市公司股东的净利润
metric_code = net_profit_parent
metric_name = 归母净利润
```

第一版至少支持以下核心指标：

```text
revenue
operating_cost
gross_profit
gross_margin
net_profit
net_profit_parent
net_profit_deducted
operating_cash_flow
total_assets
total_liabilities
equity_parent
asset_liability_ratio
eps_basic
roe_weighted
rd_expense
selling_expense
management_expense
financial_expense
inventory
accounts_receivable
```

## 5.2 难点二：表头不统一

不同财报表头可能是：

```text
项目 | 本期金额 | 上期金额
项目 | 本报告期 | 上年同期
项目 | 2026年1-3月 | 2025年1-3月
项目 | 期末余额 | 期初余额
```

解决方案：

新增 `header_mapper.py`。

它负责识别：

```text
current_period_column
previous_period_column
current_balance_column
previous_balance_column
yoy_column
```

规则示例：

```python
CURRENT_PERIOD_KEYWORDS = [
    "本期金额",
    "本报告期",
    "本期发生额",
    "2026年1-3月",
    "本期"
]

PREVIOUS_PERIOD_KEYWORDS = [
    "上期金额",
    "上年同期",
    "上期发生额",
    "2025年1-3月"
]

CURRENT_BALANCE_KEYWORDS = [
    "期末余额",
    "本报告期末",
    "2026年3月31日"
]

PREVIOUS_BALANCE_KEYWORDS = [
    "期初余额",
    "上年度末",
    "2025年12月31日"
]
```

第一版不需要覆盖所有情况，但要把无法识别的表头写入 `extraction_error`。

## 5.3 难点三：单位不统一

财报可能出现：

```text
单位：元
单位：千元
单位：万元
单位：百万元
单位：亿元
```

解决方案：

新增 `unit_normalizer.py`。

入库时保存：

```text
raw_value
unit
scale
value
```

例如：

```text
原始值：100,000
单位：万元
scale = 10000
value = 1000000000
```

SQL 中 `value` 建议统一存成元。

------

## 5.4 难点四：PDF 表格解析错误

PDF 表格可能出现：

1. 行列错位。
2. 合并单元格丢失。
3. 跨页表格断裂。
4. 负数括号解析错误。
5. 单位丢失。
6. 表头识别失败。
7. 指标重复抽取。

解决方案：

1. 每条 fact 保存来源页码、表格 ID、行名、列名。
2. 解析失败不硬塞入 financial_fact。
3. 失败项写入 `extraction_error`。
4. 未识别指标写入 `unknown_metric`。
5. 对关键指标设置 confidence。
6. 同一公司同一期间同一指标重复出现时做冲突检测。

冲突检测逻辑：

```text
如果 ticker + report_period + metric_code 出现多个不同 value：
  - 如果来自同一表且差异极小，保留一个。
  - 如果差异较大，写入 extraction_error。
  - 不要静默覆盖。
```

------

## 5.5 难点五：大模型不应该直接决定数值

LLM 可以辅助：

1. 判断用户意图。
2. 映射用户问题到 `metric_code`。
3. 对未知指标给候选映射。
4. 生成自然语言解释。

LLM 不应该：

1. 直接从 RAG 文本中猜精确数字。
2. 直接改写 SQL 中的事实。
3. 在没有来源页码时输出精确财报数字。
4. 忽略单位和报告期。

# 十三、最终项目亮点表达

后续代码和 README 应该围绕这个亮点组织：

> 本项目在原有多 Agent 金融问答系统上，新增了面向金融场景的数据与记忆架构。
> 系统将用户画像、短期缓存、文档索引保存在 KVStore；将财报核心指标结构化进入 SQLite FactStore；将研报、公告、政策、管理层讨论等长文本进入 ChromaDB RAG；通过 Query Router 判断用户问题应走 SQL、RAG、KVStore 还是 Web/API。
> 对于财报数值类问题，系统不依赖 RAG 语义猜测，而是通过结构化指标库进行精确查询，并保留原始指标名、单位、来源页码和表格信息，提升金融问答的准确性和可验证性。
