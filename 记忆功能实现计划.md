## 一、项目背景

当前 FinIntel-Multi-Agent 的记忆系统完全依赖文件 JSON + ChromaDB，缺乏高性能、可修改的实时记忆层。本计划将自研的 kvstore（类 Redis、C 语言、协程驱动、RESP 协议）集成到 Agent 记忆架构中，形成 **四层混合记忆 + 遗忘机制**。

### 1.1 kvstore 关键语义（代码已验证）

| 操作 | key 存在 | key 不存在 |
|------|---------|-----------|
| `SET` | 返回 `:0`（拒绝覆盖） | 返回 `+OK` |
| `MOD` | 返回 `+OK` | 返回 `:0`（拒绝新建） |
| `GET` | 返回值 | 返回 `$-1` |
| `DEL` | 返回 `+OK` | 返回大于0整数 |
| `EXIST` | 返回 `:1` | 返回 `:0` |

> Python 客户端需实现 `upsert(key, value)`: 先 MOD → 若失败则 SET，保证"有则改之，无则加之"。

### 1.2 kvstore 可用引擎与分配

| 引擎 | 特性 | 分配用途 |
|------|------|---------|
| Hash (HSET/HGET/HDEL) | O(1) 平均，1024桶，字段级存取 | 用户画像、股票基础信息、RAG索引 |
| SkipList (SSET/SGET/SDEL) | O(logN)，时间有序，支持 COUNT | 对话历史、操作日志、按时间排序的数据 |
| Array (SET/GET/DEL) | O(n) 线性扫，102万槽位 | 临时缓存、非高频访问数据 |

## 二、四层记忆架构

```
┌─────────────────────────────────────────────────────────────────┐
│                     MemoryAgent.retrieve(query)                  │
│                                                                  │
│   用户输入 → 实体提取 → [L1 → L2 → L3 → L4 逐层回退检索]         │
│                                                                  │
│   命中即返回 (短路逻辑)，只有上层 MISS 才查下层                   │
└──────────────────────────────────────────────────────────────────┘
          │            │            │            │
          ▼            ▼            ▼            ▼
┌──────────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐
│ L1 瞬时记忆  │ │L2 短期   │ │L3 长期   │ │L4 语义记忆   │
│ Python dict  │ │kvstore   │ │kvstore   │ │ ChromaDB     │
│ 会话级       │ │ 用户级   │ │ 公共层   │ │ 向量检索     │
│ 非持久化     │ │ 持久化   │ │ 持久化   │ │ 深度研报     │
│ <1μs         │ │ <100μs   │ │ <100μs   │ │ 10-100ms     │
└──────────────┘ └──────────┘ └──────────┘ └──────────────┘
```

### 2.1 第一层：瞬时记忆 (Transient Memory)

**定位**：会话级 · 速度最快 · 非持久化

| 维度 | 说明 |
|------|------|
| 存储位置 | Python 内存 dict（`TransientMemory._buffer`），或 kvstore 临时 namespace `temp:session:{sid}:*` |
| 生命周期 | 会话创建 → 会话关闭立即清空 |
| 容量 | 最近 10 轮对话 |
| 查询延迟 | <1μs（Python dict）/ <50μs（kvstore） |
| 持久化 | 否 |

**存什么**：
- 当前会话最近 5~10 轮对话的 query/answer 摘要
- 用户这几分钟内连续问的股票代码列表（实体追踪）
- 当前会话中用户明确的临时偏好

**数据结构** (kvstore 模式)：
```
SSET temp:session:{session_id}:conversation  {timestamp}  "{role}:{content_summary}"
HSET temp:session:{session_id}:entities     600519  "贵州茅台"  000858  "五粮液"
HSET temp:session:{session_id}:meta         created_at  "{timestamp}"  turn_count  "5"
```

**作用**：
- 代词消解："它的估值高不高" → 从 L1 查到上一轮说的"茅台"
- 对话连贯："继续分析" → 知道继续什么

### 2.2 第二层：短期用户记忆 (Short-term User Memory)

**定位**：用户级 · kvstore 主力 · 跨会话持久

| 维度 | 说明 |
|------|------|
| 存储位置 | kvstore Hash 引擎，namespace `user:{uid}:*` |
| 生命周期 | 持久化，30天未访问自动清理 |
| 容量 | 每用户最多 50 个关注股票、100 条查询摘要 |
| 查询延迟 | <100μs（本地 TCP 回环） |
| 持久化 | 是（kvstore AOF 增量 + RDB 全量） |

**存什么**：
```
# 用户画像
HSET user:{uid}:profile
    name               "张总"
    philosophy         "价值投资"
    style              "长期持有"
    experience         "10年"
    preferences        "白酒,新能源,银行"
    risk_tolerance     "中高"
    max_position_pct   "30"
    stop_loss_pct      "8"
    take_profit_pct    "25"
    preferred_sectors  "消费,科技"
    update_frequency   "每周"

# 关注股票列表
HSET user:{uid}:watchlist
    600519  "贵州茅台"
    000858  "五粮液"
    300750  "宁德时代"

# 最近查询历史摘要 (SkipList 按时间排序)
SSET user:{uid}:query_history
    20260512:140000  "{query_abstract}"
    20260512:130000  "{query_abstract}"

# 策略参数
HSET user:{uid}:strategy
    ma_short    "5"
    ma_long     "20"
    rsi_period  "14"
    volume_ratio_threshold "1.5"
```

**作用**：
- 跨窗口记住用户是谁、炒什么股、什么风格
- 不用每次都问"你持有什么股票？""你的风险偏好是什么？"
- 投资建议可以个性化："您偏好价值投资，当前茅台 PE 28.5 处于历史中低位..."

### 2.3 第三层：长期结构化记忆 (Long-term Structured Memory)

**定位**：公共层 · 股票基础数据 · 所有用户共享

| 维度 | 说明 |
|------|------|
| 存储位置 | kvstore Hash 引擎，namespace `stock:{code}:*` |
| 生命周期 | 持久化，价格数据 7 天 TTL |
| 容量 | 全 A 股约 5000 只，每只约 1KB = 5MB |
| 查询延迟 | <100μs |
| 更新频率 | 行情类 1~5 分钟刷新，基础信息按日刷新 |

**存什么**：
```
# 股票基础信息
HSET stock:600519:base
    name         "贵州茅台"
    sector       "白酒"
    industry     "食品饮料"
    market_cap   "2350000000000"
    pe_ttm       "28.5"
    pb           "9.2"
    turnover_rate "0.35"
    listing_date "2001-08-27"

# 最新行情快照（准实时，1~5分钟刷新）
HSET stock:600519:quote
    price        "1445.00"
    change_pct   "+1.25"
    volume       "2456789"
    high         "1450.00"
    low          "1430.00"
    update_time  "20260512:143000"

# RAG 入口索引（指向 ChromaDB）
HSET stock:600519:rag_index
    report_2025q4   "chroma:doc:rep_600519_2025q4"
    report_2025q3   "chroma:doc:rep_600519_2025q3"
    annual_2024     "chroma:doc:ann_600519_2024"
    research_citic  "chroma:doc:res_citic_202603"
    policy_alcohol  "chroma:doc:pol_alcohol_202602"
```

**作用**：
- O(1) 获取股票基础信息，不用每次都调 API
- RAG 入口：知道某只股票有哪些可用的研报/财报，直接定位 ChromaDB doc_id
- 行情快照加速：Agent 需要"茅台现在多少钱"时，kvstore 返回比调新浪 API 快 10 倍

### 2.4 第四层：长期语义记忆 (Semantic Memory)

**定位**：RAG 向量库独占 · 深度研报 · 不存结构化小字段

| 维度 | 说明 |
|------|------|
| 存储位置 | ChromaDB（保持现有） |
| 生命周期 | 仅保留近 1 年，旧文档归档到 `archive/` 目录 |
| 内容 | 券商研报全文、年报/季报、行业政策原文、产业链分析、策略原理 |
| 查询延迟 | 10-100ms |

**存什么**（仅限深度非结构化文本）：
- 券商研报全文（中信、华泰、招商等）
- 上市公司年报/季报/业绩说明会内容
- 行业政策原文（监管文件、行业标准）
- 产业链逻辑深度分析
- 技术指标/策略的原理和案例
- 历史行情复盘（月度/季度）

**RAG 文档元数据规范**（与 L3 索引对应）：
```json
{
  "doc_id": "chroma:doc:rep_600519_2025q4",
  "title": "贵州茅台2025年年报深度分析",
  "source": "中信证券",
  "date": "2026-03-15",
  "stock_codes": ["600519"],
  "category": "annual_report",
  "sector": "白酒",
  "in_kvstore_index": true
}
```

**作用**：
- 回答深度逻辑问题："茅台 Q4 营收结构如何？""新能源政策对锂矿影响？""均线策略原理？"
- 前三层都解决不了时才调用（短路逻辑，避免不必要的向量检索）

## 三、遗忘机制 (Forgetting Mechanism)

### 3.1 各层遗忘策略

| 层级 | 数据类型 | 遗忘策略 | 实现方式 |
|------|---------|---------|---------|
| L1 瞬时 | 会话上下文 | 会话关闭全清 | `session.close()` 触发 `flush_l1()` |
| L2 短期 | 关注股票 | 30天无查询 → 移除 | 每次检索时检查 `last_accessed`，过期 DEL |
| L2 短期 | 查询历史 | 保留最近 100 条，或 30 天滚动 | SSET COUNT>100 → DEL 最旧 |
| L3 长期 | 行情快照 | 7 天 TTL | 读取时检查 `update_time`，过期异步刷新或标记 |
| L3 长期 | 基础信息 | 按日刷新覆盖 | 每天定时任务全量刷新 |
| L3 长期 | RAG 索引 | 随 RAG 归档联动 | ChromaDB 删文档时同步 DEL L3 索引 |
| L4 语义 | 研报/财报 | 1 年保留，超期归档 | 定时任务扫描 `date`，超 1 年移入 `archive/` |

### 3.2 遗忘触发时机

```
┌──────────────────────────────────────────┐
│  触发时机            │  执行动作           │
├──────────────────────────────────────────┤
│  会话关闭            │  清空 L1            │
│  每次 MemoryAgent.   │  L2 检查 30 天      │
│  retrieve() 调用     │  过期 DEL           │
│  每次读取 L3 行情    │  检查 update_time    │
│                      │  >7天 → 异步刷新    │
│  每日凌晨 3:00       │  L3 基础信息全量    │
│  (cron)              │  刷新 + L4 归档     │
│  RAG 文档删除时      │  联动清理 L3 索引   │
└──────────────────────────────────────────┘
```

**关键设计**:
```python
class KvstoreClient:
    def __init__(self, host="127.0.0.1", port=2000):
        ...
    
    def upsert(self, key: str, value: str) -> bool:
        """有则 MOD，无则 SET"""
        resp = self.execute(f"MOD {key} {value}")
        if resp == ":0\r\n":  # key 不存在，MOD 失败
            resp = self.execute(f"SET {key} {value}")
        return resp == "+OK\r\n"
    
    def hset_multi(self, key: str, field_values: dict) -> int:
        """批量 HSET，返回成功数"""
        ...
    
    def pipeline(self, commands: list[str]) -> list[str]:
        """一次 send 多条命令，减少 RTT"""
        ...
```

```python
class KvstoreMemoryBackend:
    def __init__(self, client: KvstoreClient, namespace: str):
        self.client = client
        self.ns = namespace  # 如 "user:default"
    
    def _k(self, sub_key: str) -> str:
        """构造完整 key: user:default:profile"""
        return f"{self.ns}:{sub_key}"
    
    def store_profile(self, data: dict) -> int:
        """用户画像：HSET user:{uid}:profile field1 v1 field2 v2..."""
        return self.client.hset_multi(self._k("profile"), data)
    
    def get_profile_field(self, field: str) -> str:
        return self.client.hget(self._k("profile"), field)
```

### Phase 3: 四层记忆管理器整合

**文件**: `src/memory/hybrid_memory.py`

**功能清单**:
- [ ] `HybridMemorySystem` 统一入口
- [ ] 短路检索逻辑：L1 → L2 → L3 → L4
- [ ] `retrieve(query)` 返回四层合并结果
- [ ] 层间交互：L3 提供 `rag_doc_ids` → L4 精确检索

**检索流程**:
```python
async def retrieve(self, query: str) -> MemoryContext:
    entities = extract_entities(query)  # 600519, 茅台, 白酒
    
    # L1: 瞬时记忆 (会话上下文)
    l1_result = self.l1.retrieve_context(k=5)
    if self._is_sufficient(l1_result, query):
        return l1_result  # 短路：代词消解等简单问题
    
    # L2: 用户记忆 (偏好、关注股)
    l2_result = self.l2.retrieve(entities)
    
    # L3: 股票公共数据 + RAG 索引
    l3_result = self.l3.retrieve_stocks(entities)  # 基础信息+行情
    l3_rag_ids = self.l3.get_rag_ids(entities)      # RAG 入口索引
    
    # L4: 语义记忆 (仅当需要深度分析且前三层不足时)
    l4_result = None
    if self._needs_deep_semantic(query, l1_result, l2_result, l3_result):
        l4_result = self.l4.retrieve(query, doc_ids=l3_rag_ids)
    
    return MemoryContext(
        transient=l1_result,
        user=l2_result,
        stock=l3_result,
        semantic=l4_result,
    )
```

**验证方法**:

### Phase 4: 遗忘机制实现

**文件**: `src/memory/forgetting.py`

**功能清单**:
- [ ] `ForgettingManager` 统一管理各层遗忘
- [ ] L1: `on_session_close(session_id)` 全清
- [ ] L2: `check_and_forget_user(user_id)` 检查 30 天
- [ ] L3: `check_price_staleness(stock_code)` 检查 7 天
- [ ] L4: `archive_old_docs()` 1 年归档
- [ ] 定时任务 cron 配置

**实现要点**:
```python
class ForgettingManager:
    L2_RETENTION_DAYS = 30    # 用户关注股票
    L2_HISTORY_MAX = 100       # 查询历史上限
    L3_PRICE_TTL_DAYS = 7      # 行情快照
    L4_ARCHIVE_DAYS = 365      # 研报保留
    
    async def run_l2_forgetting(self, user_id: str):
        watchlist = self.client.hgetall(f"user:{user_id}:watchlist")
        for code, name in watchlist.items():
            last_access = self.client.hget(f"user:{user_id}:access_log", code)
            if days_since(last_access) > self.L2_RETENTION_DAYS:
                self.client.hdel(f"user:{user_id}:watchlist", code)
                logger.info(f"遗忘: 用户{user_id} 关注股 {code} {name} (30天未访问)")
    
    def on_session_close(self, session_id: str):
        """L1 遗忘：清空会话 namespace"""
        count = self.client.del_keys_by_prefix(f"temp:session:{session_id}:")
        logger.info(f"L1 遗忘: 会话{session_id} 清除 {count} 条瞬时记忆")
```

## 七、风险与注意事项

1. **kvstore SET/MOD 语义**：Python 客户端必须正确处理 `:0` 返回值，区分"键已存在"和"键不存在"
2. **kvstore 单点故障**：kvstore 挂掉时，Agent 需降级到纯 ChromaDB 模式（已设计 fallback）
3. **并发写入**：多 Agent 同时 HSET 同一 key 的不同 field 是安全的（Hash 字段级操作），但整 key SET 需注意
4. **AOF 性能影响**：记忆写操作会触发 AOF，高频写入时建议 `enable_aof=0`（通过 RDB 定时保存足够）
5. **命名空间隔离**：严格执行 `temp:/user:/stock:` 前缀规范，防止跨层污染
6. **向后兼容**：Phase 6 不改动现有 ChromaDB 逻辑，只是增加 kvstore 作为前置缓存层
